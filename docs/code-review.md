# 代码审查规则（初版）

适用目录：`src/`, `apps/`, `framework/`

## 目标

- 保证代码安全、可维护、可测试并符合团队风格。

## 必检项

- 安全：输入校验、敏感信息不泄露、避免未授权访问路径、使用安全的随机/加密库。
- 正确性：边界条件、错误处理、资源释放、并发竞态检查。
- 可维护性：函数/类职责单一、函数长度适中、清晰命名、注释必要的复杂逻辑。
- 可测试性：核心逻辑有单元测试、避免难以测试的全局状态、易于注入依赖。
- 性能：避免显著的 N^2/内存泄漏，热点路径提供基准并标注优化点。
- 风格：遵循项目现有编码规范（命名、缩进、注释风格），通过 linter/format 自动化检查。

## 推荐实践与示例

- 小函数优于大函数：函数长度建议 < 200 行，单一职责。
- 错误处理统一：使用项目约定的错误/异常处理模式。
- 资源管理：优先使用 RAII 或智能指针，明确所有权边界。

示例：

```cpp
// 好：清晰的职责与错误处理
Result DoWork(Input in) {
    if (!Validate(in)) return Result::Err("invalid input");
    auto res = Worker::Process(in);
    return res;
}
```

## PR 检查清单（可嵌入 PR 模板）

- [ ] 变更说明与目的清晰
- [ ] 涉及 API/ABI 变更已注明兼容性影响
- [ ] 单元/集成测试已添加或更新
- [ ] 静态检查（linter/formatter）通过
- [ ] 性能/内存影响已评估（必要时附基准）
- [ ] 安全审查已完成（涉及凭证/加密/输入时）

## 建议流程

1. 作者在 PR 描述中填写“检查清单”项。
2. reviewer 针对必检项逐条确认并填写意见。
3. 若为重大变更，建议创建讨论 issue 并在 PR 中引用。
